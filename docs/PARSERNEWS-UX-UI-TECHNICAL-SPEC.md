# Техническое задание на UX/UI дизайн

## 1. Краткое описание продукта

| Параметр | Описание |
| --- | --- |
| Название проекта | **ParserNews** |
| Тип продукта | Веб-панель управления контентным агрегатором и пайплайном парсинга |
| Платформа | **Web, Desktop-first, adaptive** |
| Технический стек | Frontend: **React + TypeScript**. Backend: **Node.js + Express + TypeScript**. Инфраструктура: **PostgreSQL, Redis, BullMQ, Keycloak** |
| Целевая аудитория | Контент-менеджеры, аналитики, операторы данных, технические специалисты, администраторы |
| Основная бизнес-цель | Сократить время на ручной сбор и контроль данных из внешних источников, дать единый интерфейс для запуска парсеров, проверки очередей, управления статьями, загрузки VPO-файлов и контроля доступов |

### Основные функции MVP

1. Авторизация и ролевая модель через Keycloak / API key.
2. Дашборд и запуск парсеров вручную.
3. Мониторинг очередей обработки и ошибок.
4. Просмотр и редактирование данных: статьи, компании, теги.
5. Загрузка и управление VPO-сводами.

### Бизнес-ценность интерфейса

- Свести рабочие действия оператора в одну панель без перехода по нескольким системам.
- Сделать состояние пайплайна прозрачным и быстро диагностируемым.
- Дать безопасный и понятный контур администрирования: роли, ошибки, подтверждения опасных действий.

### 2.2. Логическая вложенность и навигация

| Уровень | Раздел | Назначение |
| --- | --- | --- |
| 1 | Авторизация | Вход в систему и проверка доступа |
| 1 | Главная | Точка входа после логина, быстрые действия |
| 1 | Операционные разделы | Статьи, Компании, Теги, Очереди, Настройки, VPO |
| 1 | Пользовательские разделы | Личный кабинет |
| 1 | Администрирование | Доступ и роли |

### 2.3. Навигационная модель

- Основная навигация: левый sidebar.
- Верхний уровень: плоская структура без глубоких вложенных меню.
- Внутри разделов: фильтры, табличные представления, локальные действия.
- Доступность разделов зависит от ролей и permissions.

### 2.4. Принципы IA

- Все основные рабочие сценарии должны быть достижимы максимум за 2 клика от главной.
- Опасные действия не должны смешиваться с просмотровыми.
- Разделы “данные”, “настройки” и “администрирование” должны быть визуально разделены.
- Система должна показывать не только результат, но и контекст: источник, состояние, очередь, право доступа.

## 3. User Flow

### 3.1. Ключевой User Flow: от входа до запуска парсинга и проверки результата

| Шаг | Действие пользователя | Ожидание пользователя | Реакция системы |
| --- | --- | --- | --- |
| 1 | Открывает приложение | Понять, нужен ли вход | Показать экран авторизации и текущий статус системы |
| 2 | Входит через Keycloak | Быстро попасть в рабочую область | Редирект на главную после успешной авторизации |
| 3 | Видит дашборд | Понять текущее состояние сервиса | Показать доступные действия, статус системы, описание источников |
| 4 | Нажимает запуск парсера | Отправить задачу в пайплайн | Показать toast-успех, обновить состояние очередей |
| 5 | Переходит в раздел “Очереди” | Увидеть этапы обработки | Показать parse/adapt/store, количество waiting/active/completed/failed |
| 6 | При необходимости открывает “Статьи” | Проверить результат парсинга | Показать новые записи, фильтры, поиск |
| 7 | Включает режим редактирования | Исправить категорию или удалить мусорные записи | Показать inline-редактирование и подтверждающие действия |

### 3.2. Альтернативные сценарии

| Сценарий | Что должен делать интерфейс |
| --- | --- |
| Нет прав на раздел | Показать понятный экран “Доступ ограничен” с указанием недостающего permission |
| Очередь упала | Показать failed jobs и диагностический контент без технического шума |
| Нет данных после запуска | Показать пустой экран с объяснением, что делать дальше |
| Истёк токен Keycloak | Показать сообщение о завершении сессии и CTA на повторный вход |
| Ошибка сервера | Показать error-state на уровне экрана, не ломая layout |

### 3.3. User Flow для администратора ролей

1. Авторизоваться как пользователь с правами администрирования.
2. Открыть раздел “Доступ и роли”.
3. Найти пользователя по логину / email / имени.
4. Изменить роль пользователя.
5. Сохранить изменения.
6. Получить подтверждение об успешном обновлении.

## 4. UX-требования

### 4.1. Сетка и layout

| Элемент | Требование |
| --- | --- |
| Базовый layout | Sidebar + основная рабочая область |
| Основная сетка | 12 колонок в контентной зоне |
| Ширина контента | Рабочий desktop breakpoint: 1280–1440 px |
| Минимальная поддерживаемая ширина | 1024 px без потери критичных функций |
| Адаптивность | На tablet/mobile sidebar превращается в off-canvas |
| Шаг вертикального ритма | 8 px |

### 4.2. Навигация

- Sidebar должен быть постоянным на desktop.
- Активный раздел должен быть визуально очевиден.
- В подвале sidebar должна оставаться системная информация: health, пользователь, роль, auth-state.
- Верхняя часть каждого экрана должна содержать:
  - заголовок раздела
  - подзаголовок/описание
  - контекстные CTA

### 4.3. Принципы UX-поведения

- Каждое действие должно иметь мгновенную обратную связь.
- Опасные действия требуют явного режима или подтверждения.
- Пользователь должен понимать, что можно делать на экране, без чтения документации.
- На таблицах обязателен предсказуемый паттерн: фильтр -> таблица -> пагинация.

### 4.4. Состояния интерфейса

#### Пустые состояния

- Нет статей.
- Нет упавших задач.
- Нет результатов поиска.
- Нет истории VPO.
- Нет пользователей в фильтре ролей.

Для каждого empty-state дизайнер должен предусмотреть:

- заголовок состояния
- короткое объяснение
- понятный CTA или совет, что делать дальше

#### Ошибки

- Ошибка авторизации
- Ошибка загрузки экрана
- Ошибка сохранения формы
- Ошибка rate-limit
- Ошибка сетевого соединения
- Недостаточно прав

Требования:

- ошибки не должны “ломать” layout
- текст ошибки должен быть отделён от технических логов
- должен быть CTA: “повторить”, “обновить”, “войти заново” или “вернуться”

#### Загрузка

- skeleton или loading-state на уровне экрана
- inline-loading на уровне кнопки действия
- фоновое обновление polling-сценариев без мигающего full-screen loader

#### Успешные действия

- toast/notice после запуска парсера, сохранения роли, изменения категории, загрузки VPO и удаления
- успех не должен блокировать следующий шаг пользователя

### 4.5. Требования к таблицам

- фиксированная логика: фильтры сверху, таблица ниже, пагинация внизу
- текст в ячейках должен переноситься или обрезаться предсказуемо
- в title-ячейках допустим второй слой информации: summary/подпись
- действия по строке должны быть доступны, но не шуметь в default state
- для destructive действий желательно использовать режим редактирования или явные danger-кнопки

### 4.6. Accessibility

- Контраст не ниже WCAG AA для основного текста и controls.
- Все controls должны иметь hover/focus/disabled states.
- Таблицы и формы должны быть управляемы с клавиатуры.
- Ошибки форм должны быть привязаны к конкретному полю.

## 5. UI-требования

### 5.1. Стилистика

Рекомендованное направление: **строгий operational business UI**.

Характер интерфейса:

- профессиональный
- спокойный
- точный
- “инструментальный”, а не маркетинговый
- ориентирован на длительную работу в интерфейсе

### 5.2. Визуальные принципы

| Направление | Требование |
| --- | --- |
| Визуальный стиль | Строгий business + light operational UI |
| Эмоциональный тон | Контроль, надёжность, прозрачность |
| Насыщенность | Средняя, без декоративной перегрузки |
| Иерархия | Явное разделение между primary actions, info blocks, warning/danger zones |

### 5.3. Цветовая система

| Назначение | Рекомендация |
| --- | --- |
| Основной фон | Светлый нейтральный |
| Карточки и таблицы | Белый / почти белый |
| Primary accent | Холодный синий или индиго |
| Success | Зелёный |
| Warning | Янтарный |
| Danger | Красный |
| Secondary text | Серо-нейтральный |
| Sidebar | Тёмный контрастный блок |

Важно:

- не использовать кислотные или “игровые” цвета
- не перегружать экраны множеством конкурентных акцентов
- цвета статусов должны быть едины во всех разделах

### 5.4. Типографика

| Элемент | Требование |
| --- | --- |
| UI font | Современный гротеск, оптимизированный под интерфейсы |
| Код / ID / тех. поля | Моноширинный шрифт |
| Заголовок экрана | 28–32 px |
| Заголовок секции | 18–22 px |
| Основной текст | 14–16 px |
| Подписи и метки | 12–13 px |

### 5.5. Визуальная иерархия

- Заголовок страницы должен быть самым заметным элементом верхней зоны.
- CTA первого порядка: запуск, сохранить, применить.
- CTA второго порядка: обновить, сбросить, отменить.
- Danger actions должны быть визуально отделены.
- Системная информация должна быть заметной, но не доминирующей.

### 5.6. Адаптивность

Дизайнер должен подготовить макеты минимум для:

- 1440 px desktop
- 1280 px рабочий desktop
- 1024 px compact desktop/tablet
- 768 px mobile/tablet fallback для навигации и таблиц

## 9. Вопросы и допущения

### 9.1. Допущения для текущей версии ТЗ

- Проект = текущий продукт **ParserNews**, а не абстрактный шаблон.
- В приоритете рабочий desktop-интерфейс для внутренних пользователей.
- Ролевая модель уже существует и должна быть отражена визуально.
- Фокус дизайна — не on-boarding, а ежедневная эксплуатация системы.

### 9.2. Вопросы, которые желательно уточнить до старта дизайна

1. Нужен ли отдельный бренд-гайд или текущего product-neutral business style достаточно?
2. Должны ли данные таблиц поддерживать экспорт, сортировки и сохранённые фильтры в следующем релизе?
3. Требуется ли отдельный mobile-first сценарий для операторов вне офиса или достаточно adaptive fallback?
4. Нужны ли детальные страницы сущностей в будущем: карточка статьи, карточка компании, история изменения роли?
5. Нужен ли dark theme как roadmap-функция?

### 9.3. Критерии приёмки дизайна

- Все экраны MVP отрисованы в desktop и ключевых adaptive-вариантах.
- Для каждого экрана есть loading / empty / error states.
- Все опасные действия имеют понятный UX-паттерн.
- UI Kit покрывает не менее 90% используемых компонентов приложения.
- Макеты пригодны для прямой передачи в frontend-разработку без устных пояснений.

## 10. Итог для дизайнера

Нужно спроектировать **строгий, адаптивный operational UI** для системы управления пайплайном парсинга и контентом. Главный акцент — на скорости выполнения рабочих задач, прозрачности состояния системы и безопасной работе с данными и правами. Интерфейс должен быть дружелюбен к ежедневной эксплуатации, а не выглядеть как демонстрационный dashboard.
